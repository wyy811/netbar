## 第一部分：Go + Gin 后端核心知识

### 1.1 Gin 框架基础

#### 路由系统
| 知识点    | 项目场景             | 关键理解                 |
| --------- | -------------------- | ------------------------ |
| 路由分组  | `/api/v1` 统一前缀   | 用 `r.Group()` 组织模块  |
| 路径参数  | `:id` 动态段         | `c.Param("id")` 获取     |
| 查询参数  | `?keyword=xxx`       | `c.Query("keyword")`     |
| 表单参数  | POST 表单提交        | `c.PostForm("username")` |
| JSON 绑定 | 请求体 JSON → 结构体 | `c.ShouldBindJSON(&req)` |

**📌 常见陷阱**：
- `ShouldBindJSON` 只能调用一次，body 被读取后无法再次绑定
- 结构体字段必须大写导出，否则无法绑定

---

#### 上下文（Context）使用
```
c *gin.Context 包含：
- 请求信息（Header/Params/Body）
- 响应方法（JSON/HTML/File）
- 数据传递（Set/Get）——在中间件间共享数据
- 终止链式调用（Abort）
```

**💡 最佳实践**：
- 在中间件中用 `c.Set("user_id", uid)`，Handler 中用 `c.Get("user_id")`
- 业务逻辑不要直接依赖 `*gin.Context`，应解耦到 Service 层

---

#### 中间件原理
```
执行顺序：请求 → 全局中间件 → 路由组中间件 → 具体路由处理 → 响应
```
**需要实现的中间件**：
1. **CORS**：处理跨域，响应 OPTIONS 预检
2. **JWT 验证**：提取 Token，验证后存入上下文
3. **日志记录**：记录请求路径、耗时、状态码
4. **限流**：基于 IP 或用户 ID 的访问频率限制
5. **错误恢复**：捕获 panic，返回 500 而非崩溃

**📌 关键**：中间件里必须调用 `c.Next()` 或 `c.Abort()`，否则请求会挂起。

---

#### 响应封装
统一响应结构可以大幅减少重复代码。实现思路：
- 定义标准结构体 `{code, message, data}`
- 封装辅助函数：`Success(c, data)`、`Error(c, code, msg)`
- 错误码分类：0=成功，1xxx=参数错误，2xxx=认证错误，3xxx=业务错误

### 1.2 GORM ORM 框架

#### 模型定义标签
```go
type User struct {
    ID        uint      `gorm:"primaryKey"`
    Username  string    `gorm:"uniqueIndex;size:50;not null"`
    Password  string    `gorm:"column:password_hash"`  // 自定义列名
    CreatedAt time.Time `gorm:"autoCreateTime"`
    UpdatedAt time.Time `gorm:"autoUpdateTime"`
}
```
**常用标签**：
- `column`：指定数据库列名
- `type`：指定列类型（如 `decimal(10,2)`）
- `default`：默认值
- `index` / `uniqueIndex`：创建索引
- `-`：忽略该字段（不存数据库）

---

#### CRUD 操作模式
| 操作     | GORM 方法                                | 项目场景             |
| -------- | ---------------------------------------- | -------------------- |
| 创建     | `db.Create(&user)`                       | 新增机位/会员        |
| 查询单条 | `db.First(&user, id)`                    | 登录验证             |
| 查询列表 | `db.Where("status = ?", 1).Find(&users)` | 机位列表             |
| 更新     | `db.Model(&user).Update("balance", 100)` | 会员充值             |
| 删除     | `db.Delete(&computer, id)`               | 删除机位（需软删除） |

**📌 陷阱**：
- `First` 没找到记录会返回 `ErrRecordNotFound`，需要单独处理
- `Updates` 默认不更新零值字段，需用 `Select` 指定或使用 map

---

#### 高级查询（本项目高频）
| 需求       | GORM 写法                                                    |
| ---------- | ------------------------------------------------------------ |
| 模糊搜索   | `Where("name LIKE ?", "%"+keyword+"%")`                      |
| 时间范围   | `Where("created_at BETWEEN ? AND ?", start, end)`            |
| 关联预加载 | `Preload("Member").Preload("Computer").Find(&sessions)`      |
| 分组统计   | `Select("area, count(*)").Group("area")`                     |
| 原生 SQL   | `Raw("SELECT * FROM computers WHERE ...")`                   |
| 子查询     | `Where("price > (?)", db.Table("rates").Select("avg(price)"))` |

---

#### 事务处理（关键！）
上机操作涉及三张表，必须用事务：

```
db.Transaction(func(tx *gorm.DB) error {
    // 1. 锁定机位行（悲观锁）
    tx.Clauses(clause.Locking{Strength: "UPDATE"}).First(&computer, id)
    // 2. 创建 session 记录
    tx.Create(&session)
    // 3. 更新 computer 状态
    tx.Model(&computer).Update("status", 1)
    // 返回 nil 提交，返回 error 回滚
})
```

**💡 两种锁**：
- **悲观锁**：`SELECT FOR UPDATE`，适合高并发争抢同一资源（机位）
- **乐观锁**：通过 version 字段，适合会员余额更新

---

#### 连接池配置
```go
sqlDB, _ := db.DB()
sqlDB.SetMaxIdleConns(10)      // 最大空闲连接
sqlDB.SetMaxOpenConns(100)     // 最大打开连接
sqlDB.SetConnMaxLifetime(time.Hour)
```
**📌**：连接数不是越大越好，需根据 MySQL 的 `max_connections` 调整。

### 1.3 JWT 认证

#### Token 结构
```
Header: { "alg": "HS256", "typ": "JWT" }
Payload: { "user_id": 1, "role": 2, "exp": 1234567890 }
Signature: HMACSHA256(header + "." + payload, secret)
```

**本项目双 Token 设计**：
- **Access Token**：短期（24h），用于 API 鉴权
- **Refresh Token**：长期（7天），用于刷新 Access Token
- Refresh Token 存储在数据库或 Redis，支持主动撤销

---

#### 中间件验证流程
```
1. 从 Authorization Header 提取 "Bearer <token>"
2. 解析 JWT，验证签名和过期时间
3. 从 claims 中提取 user_id，存入上下文
4. 检查用户状态（是否被禁用）
5. 可选：检查 Token 是否在黑名单（登出时加入）
```

**📌 陷阱**：
- JWT 一旦签发就无法修改，用户禁用后需通过黑名单方案解决
- 时间比较使用 `time.Now().Unix()`，注意时区问题

---

#### 密码加密（bcrypt）
```go
// 加密
hashed, _ := bcrypt.GenerateFromPassword([]byte(password), bcrypt.DefaultCost)

// 验证
err := bcrypt.CompareHashAndPassword(hashed, []byte(password))
```
**💡**：成本因子建议 10-12，太高影响性能，太低不安全。

### 1.7 项目分层架构

```
Handler（控制器）
    ↓ 调用
Service（业务逻辑）
    ↓ 调用
Repository（数据访问）
    ↓ 操作
Database（MySQL）
```

**职责边界**：

```
DB → Repository → Service → Handler → Router
```

## 第二部分：Vue3 前端核心知识

### 2.1 Composition API（核心语法）

#### setup 语法糖
```vue
<script setup>
import { ref, reactive, computed, watch, onMounted } from 'vue'

// 响应式数据
const count = ref(0)          // 基础类型
const user = reactive({       // 对象类型
  name: '', age: 0
})

// 计算属性
const double = computed(() => count.value * 2)

// 侦听器
watch(count, (newVal) => {
  console.log('count changed:', newVal)
})

// 生命周期
onMounted(() => {
  fetchData()
})

// 暴露给模板
defineExpose({ count })      // 父组件可通过 ref 访问
</script>
```

**📌 关键区别**：
- `ref` 用于任意值，访问需 `.value`
- `reactive` 只能用于对象，直接访问属性
- 解构 `reactive` 会丢失响应性，可用 `toRefs`

---

#### 生命周期映射
| Vue2      | Vue3 Composition API |
| --------- | -------------------- |
| created   | 直接写在 setup() 中  |
| mounted   | onMounted            |
| updated   | onUpdated            |
| destroyed | onUnmounted          |

**💡**：数据请求通常在 `onMounted` 中发起，而非 `setup` 中。

---

### 2.2 响应式原理（理解即可）

**核心机制**：
- `ref`：通过 `value` 属性的 getter/setter 拦截
- `reactive`：通过 Proxy 代理整个对象
- 依赖收集：当响应式数据被读取时，记录当前正在执行的 effect
- 派发更新：数据变化时，触发所有依赖该数据的 effect 重新执行

**📌 常见问题**：
- 数组操作：`arr[0] = xxx` 无法触发更新，应使用 `arr.splice` 或重新赋值
- 对象新增属性：使用 `obj.newProp = xxx` 无法触发，应使用 `Object.assign(obj, { newProp: xxx })`

---

### 2.3 组件通信

| 方式             | 适用场景     | 语法                                  |
| ---------------- | ------------ | ------------------------------------- |

### 2.5 状态管理（Pinia）

#### Store 定义（Options Store 风格）
```javascript
export const useAuthStore = defineStore('auth', {
  state: () => ({
    token: null,
    user: null,
    permissions: []
  }),
  getters: {
    isAdmin: (state) => state.user?.role >= 2,
    isLoggedIn: (state) => !!state.token
  },
  actions: {
    async login(username, password) {
      const res = await api.login(username, password)
      this.token = res.data.token
      this.user = res.data.user
      localStorage.setItem('token', this.token)
    },
    logout() {
      this.token = null
      this.user = null
      localStorage.removeItem('token')
    }
  }
})
```

#### Setup Store 风格（更灵活）
```javascript
export const useComputerStore = defineStore('computer', () => {
  const computers = ref([])
  const loading = ref(false)
  
  const fetchComputers = async () => {
    loading.value = true
    computers.value = await api.getComputers()
    loading.value = false
  }
  
  const getById = (id) => computers.value.find(c => c.id === id)
  
  return { computers, loading, fetchComputers, getById }
})
```

**📌 关键**：
- Store 的 action 可以是异步的（async/await）
- 在组件中使用 store 时，**不要解构** state，否则失去响应性
- 如需解构，使用 `storeToRefs(store)`

### 2.6 API 请求层（Axios 封装）

#### 实例配置
```javascript
const client = axios.create({
  baseURL: import.meta.env.VITE_API_BASE_URL,
  timeout: 10000,
  withCredentials: true
})
```

#### 请求拦截器（加 Token）
```javascript
client.interceptors.request.use(config => {
  const token = useAuthStore().token
  if (token) {
    config.headers.Authorization = `Bearer ${token}`
  }
  return config
})
```

#### 响应拦截器（统一处理错误 + Token 刷新）
```javascript
client.interceptors.response.use(
  response => response.data,
  async error => {
    if (error.response?.status === 401) {
      // Token 过期，尝试刷新
      const newToken = await refreshToken()
      if (newToken) {
        error.config.headers.Authorization = `Bearer ${newToken}`
        return client.request(error.config)  // 重试原请求
      } else {
        // 刷新失败，跳转登录
        useAuthStore().logout()
        router.push('/login')
      }
    }
    return Promise.reject(error)
  }
)
```

**📌 需要处理的边缘情况**：
- 并发请求时多个 401 会导致多次刷新，需要加一个标志位（`isRefreshing`）
- 刷新期间的其他请求应进入队列等待

### 2.7 组合式函数（Composables）

**封装可复用逻辑**：

```javascript
// composables/usePolling.js
export function usePolling(fn, interval = 5000) {
  const timer = ref(null)
  const isActive = ref(true)
  
  const start = () => {
    if (timer.value) return
    timer.value = setInterval(() => {
      if (isActive.value) fn()
    }, interval)
  }
  
  const stop = () => {
    clearInterval(timer.value)
    timer.value = null
  }
  
  onUnmounted(stop)
  return { start, stop, isActive }
}
```

**本项目可封装的 Composables**：
- `usePolling`：机位状态轮询
- `useTimer`：上机计时显示
- `useTablePagination`：表格分页与筛选逻辑
- `useDebounce`：搜索框防抖

## 第三部分：全栈联调核心知识

### 3.1 CORS 跨域

**产生原因**：浏览器同源策略，前端 `localhost:5173` 请求后端 `localhost:8080` 时触发。

**Gin 后端解决方案**：
```go
r.Use(cors.New(cors.Config{
    AllowOrigins:     []string{"http://localhost:5173"},
    AllowMethods:     []string{"GET", "POST", "PUT", "DELETE", "OPTIONS"},
    AllowHeaders:     []string{"Authorization", "Content-Type"},
    ExposeHeaders:    []string{"Content-Length"},
    AllowCredentials: true,
    MaxAge:           12 * time.Hour,
}))
```

**📌 陷阱**：
- 生产环境 `AllowOrigins` 不要用 `*`，必须指定具体域名
- 携带 Cookie 时必须设置 `AllowCredentials: true` 且 `AllowOrigins` 不能为 `*`

### 3.2 请求调试技巧

**后端调试**：
- 使用 `c.GetRawData()` 查看原始请求体（当绑定失败时）
- Gin 的 `Logger` 中间件自动打印请求路径和耗时
- 使用 Postman / Apifox 先验证接口，再联调前端

**前端调试**：
- 浏览器 Network 面板查看请求/响应详情
- Vue Devtools 查看组件状态和 Pinia Store
- Console 输出 `console.log` 或使用 `debugger` 断点

**💡 高效调试法**：
1. 先确认请求发出（Network 有记录）
2. 确认响应状态码（200/4xx/5xx）
3. 检查响应数据和前端期望是否一致
4. 检查 Pinia Store 是否正确更新

### 3.3 错误码设计（前后端约定）

| 错误码范围  | 含义       | 前端处理                        |
| ----------- | ---------- | ------------------------------- |
| 0           | 成功       | 正常展示                        |
| 10001-10099 | 参数错误   | 提示具体字段错误                |
| 20001-20099 | 未认证     | 跳转登录                        |
| 20003       | Token 过期 | 尝试刷新 Token                  |
| 30001-30099 | 业务错误   | 提示用户（余额不足/机位占用等） |
| 50000-50099 | 服务器错误 | 提示"系统异常，请稍后重试"      |

**前端统一处理**：
```javascript
if (response.code !== 0) {
  if (response.code === 20001) {
    store.logout()
    router.push('/login')
  } else {
    ElMessage.error(response.message)
  }
}
```

### 3.4 数据流转完整链路（以上机为例）

```
1. 前端：用户点击机位 → 触发 handleStart()
2. 前端：调用 sessionStore.startSession(computerId, memberId)
3. 前端：sessionStore 内部调用 api.startSession(data)
4. 前端：Axios 发送 POST /api/v1/sessions
5. 后端：Router 匹配路由 → Auth 中间件验证 Token
6. 后端：Handler 接收请求 → ShouldBindJSON 绑定参数
7. 后端：Handler 调用 Service.StartSession(computerId, memberId)
8. 后端：Service 开启事务 → 查询机位 → 创建 Session → 更新机位状态
9. 后端：Service 返回 Session 对象
10. 后端：Handler 返回 JSON {code:0, data:session}
11. 前端：Axios 解析响应 → sessionStore 更新状态
12. 前端：计算机位网格自动刷新（响应式）
13. 前端：ElMessage 显示"上机成功"
```

**📌 理解这个链路是调试的基础**，出问题时逐环节排查。

## 第四部分：数据库核心知识（与本项目相关）

### 4.1 索引设计原则

**本项目已设计索引的位置**：
- `users.username`：登录查询（唯一索引）
- `members.card_number`：会员卡号查询（唯一索引）
- `members.phone`：电话查询
- `computers.machine_number`：机位编号查询
- `computers.status`：筛选空闲/使用中机位
- `sessions.start_time`：统计报表按时间范围查询
- `sessions.computer_id`：查某台机器的历史记录
- `sessions.status`：筛选进行中的会话
- `recharges.member_id`：查询会员充值历史
- `audit_logs.created_at`：日志按时间查询

**索引选择原则**：
- 频繁出现在 `WHERE` 条件的列
- 频繁用于 `JOIN` 的关联列
- 用于 `ORDER BY` 的列
- 区分度高的列优先（如状态字段区分度低，一般不单独建索引）

**📌 索引不是越多越好**，每个索引都会降低写入性能。

---

### 4.2 事务与锁

**事务隔离级别**（MySQL 默认 REPEATABLE-READ）：
- 本项目需要避免：**不可重复读**（同一事务中两次查询结果不同）
- 上机操作使用事务保证原子性

**悲观锁示例**（防止同一机位被同时上机）：
```sql
SELECT * FROM computers WHERE id = 1 FOR UPDATE;
```
GORM 写法见 1.2 节。

**乐观锁示例**（会员余额更新）：
```sql
UPDATE members 
SET balance = balance - 10, version = version + 1 
WHERE id = 1 AND version = 5;
```

**💡 选择建议**：
- 机位状态：悲观锁（争抢激烈）
- 会员余额：乐观锁（冲突较少）

---

### 4.3 时间处理

**📌 最常踩的坑**：时区问题

```go
// GORM 配置
dsn := "root:pass@tcp(127.0.0.1:3306)/db?charset=utf8mb4&parseTime=True&loc=Local"
// loc=Local 让数据库时间转换为本地时区
```

**前端时间处理**：
```javascript
// 后端返回 ISO 8601 格式 (2026-06-28T14:30:00+08:00)
const date = new Date(response.data.created_at)
// 使用 dayjs 格式化
import dayjs from 'dayjs'
dayjs(date).format('YYYY-MM-DD HH:mm')
```

**业务时间计算**：
- 上机时长：`end_time - start_time` 计算分钟数
- 计费规则时间段：用 `TIME` 类型，只存时分

## 第五部分：部署与运维核心知识

### 5.1 Docker 基础

**Dockerfile 关键指令**：
```dockerfile
FROM golang:1.21-alpine AS builder
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -o server ./cmd/server

FROM alpine:latest
COPY --from=builder /app/server /server
EXPOSE 8080
CMD ["/server"]
```

**多阶段构建**：减少最终镜像体积（Go 二进制 + Alpine 基础镜像约 15MB）

**📌**：前端 Dockerfile 使用 Nginx 托管静态文件，不需要 Node 环境。

## 第六部分：调试排错指南

### 6.1 后端常见错误

| 错误现象                  | 可能原因             | 排查方法                            |
| ------------------------- | -------------------- | ----------------------------------- |
| 404 Not Found             | 路由未注册或路径错误 | 检查 `router.go` 注册，打印所有路由 |
| 400 Bad Request           | JSON 绑定失败        | 检查请求体格式和结构体字段          |
| 401 Unauthorized          | Token 缺失/过期/无效 | 检查 Authorization Header 格式      |
| 500 Internal Server Error | 空指针/数据库错误    | 查看日志文件，找 panic 堆栈         |
| 数据库连接超时            | 连接池耗尽/网络问题  | 检查 `max_open_conns` 和 MySQL 状态 |
| 死锁                      | 事务顺序不一致       | 检查事务中的表操作顺序              |

---

### 6.2 前端常见错误

| 错误现象     | 可能原因              | 排查方法                     |
| ------------ | --------------------- | ---------------------------- |
| 数据不更新   | 响应式丢失            | 检查是否解构了 reactive      |
| 跨域报错     | CORS 未配置           | 检查后端 CORS 中间件         |
| Token 未发送 | 拦截器未执行          | 检查 axios 实例是否使用正确  |
| 路由跳转白屏 | 路由守卫死循环        | 检查 `next()` 是否有条件错误 |
| 组件渲染异常 | 访问 undefined 的属性 | 使用 `v-if` 或可选链 `?.`    |
| 内存泄漏     | 未清除定时器/事件监听 | 检查 `onUnmounted` 清理逻辑  |
