# 网吧计费管理系统 - 全栈开发计划

## 一、项目概述

### 1.1 项目定位
面向中小型网吧的计费管理后台，支持：
- 电脑机位管理（上机/下机/预约）
- 会员充值消费管理
- 计费规则配置（时段/通宵/包时）
- 营业数据统计
- 员工操作日志

### 1.2 技术约束
- 后端：Go 1.21+ / Gin / GORM / MySQL 8.0
- 前端：Vue 3 (Composition API) / Vite / Pinia / Element Plus / Axios
- 认证：JWT (Access + Refresh Token)
- 部署：Docker Compose

### 1.3 开发顺序建议
> **阶段一**（第1-3天）：用户认证 + 机位管理  
> **阶段二**（第4-6天）：上机/下机 + 计费引擎  
> **阶段三**（第7-9天）：会员充值 + 消费记录  
> **阶段四**（第10-12天）：统计报表 + 操作日志

## 二、数据库设计（MySQL）

### 2.1 ER 图核心关系
```
用户 (user) 1---N 上机记录 (session)
会员 (member) 1---N 上机记录 (session)
会员 (member) 1---N 充值记录 (recharge)
机位 (computer) 1---N 上机记录 (session)
用户 (user) 1---N 操作日志 (audit_log)
计费规则 (rate_rule) 1---N 上机记录 (session) [通过计算时引用]
```

### 3.1 分层职责说明

| 层级           | 职责                                    | 依赖方向       |
| -------------- | --------------------------------------- | -------------- |
| **Handler**    | 参数校验、调用 Service、返回 JSON 响应  | → Service      |
| **Service**    | 核心业务逻辑、事务管理、调用 Repository | → Repository   |
| **Repository** | 数据库 CRUD、复杂查询                   | → Domain/Model |
| **Domain**     | 纯数据结构定义（无任何依赖）            | 独立           |
| **Pkg**        | 通用工具（JWT/加密/计费计算）           | 被各层调用     |

### 3.2 关键包设计要点

- **计费引擎（billing/calculator.go）**：输入开始时间、结束时间、机位ID，查询匹配的计费规则，返回总费用和明细。需处理跨时段、通宵场、会员折扣。
- **事务处理**：上机操作需同时更新 `sessions` 表插入记录 + `computers` 表状态 + `members` 表（如果是会员）。使用 GORM 的 `Transaction`。
- **并发控制**：同一机位不能同时被两人上机，需用数据库行锁或 Redis 分布式锁。

### 4.2 状态管理要点

- **auth.store**：存储 token、userInfo、权限列表。提供 `login()`、`logout()`、`refreshToken()` 方法。
- **computer.store**：存储机位列表、当前选中机位。提供 `fetchComputers()`（轮询更新）。
- **session.store**：存储当前上机记录列表。提供 `startSession()`、`endSession()` 并自动刷新机位状态。

### 4.3 组件通信策略

- **父子**：Props + Emits
- **跨层级**：Pinia Store
- **全局**：通过 Provide/Inject 注入全局配置（如 API_BASE_URL）

## 五、API 接口设计（RESTful）

### 5.1 认证授权组
```
POST   /api/v1/auth/login          # 登录（返回 access_token + refresh_token）
POST   /api/v1/auth/logout         # 登出（将 token 加入黑名单）
POST   /api/v1/auth/refresh        # 刷新 Access Token
GET    /api/v1/auth/profile        # 获取当前用户信息
PUT    /api/v1/auth/password       # 修改密码
```

### 5.2 机位管理组
```
GET    /api/v1/computers           # 列表（支持 area/status 筛选）
GET    /api/v1/computers/:id       # 详情
POST   /api/v1/computers           # 新增机位
PUT    /api/v1/computers/:id       # 修改机位（价格/区域等）
DELETE /api/v1/computers/:id       # 删除机位（仅当空闲）
PATCH  /api/v1/computers/:id/status # 变更状态（维护/启用）
```

### 5.3 上机会话组（核心）
```
GET    /api/v1/sessions            # 上机记录列表（分页 + 筛选）
GET    /api/v1/sessions/current    # 当前所有进行中的上机
GET    /api/v1/sessions/:id        # 单条记录详情
POST   /api/v1/sessions            # 上机（传入 computer_id + member_id 可选）
PUT    /api/v1/sessions/:id/end    # 下机（计算费用，更新状态）
GET    /api/v1/sessions/:id/bill   # 获取当前计费明细（预览）
```

### 5.4 会员管理组
```
GET    /api/v1/members             # 会员列表（支持 card_number/phone 搜索）
GET    /api/v1/members/:id         # 会员详情（含最近消费记录）
POST   /api/v1/members             # 注册会员
PUT    /api/v1/members/:id         # 修改会员信息（姓名/折扣等级）
PATCH  /api/v1/members/:id/status  # 冻结/解冻
```

## 六、核心业务流程设计

### 6.1 上机流程
1. 员工选择空闲机位（或扫描会员卡）
2. 系统校验机位状态（空闲）
3. 如果是会员，校验余额是否充足（预扣机制或后扣）
4. 创建 `sessions` 记录，状态为 `进行中`
5. 更新 `computers.status = 1`，记录 `current_session_id`
6. 返回上机成功信息（含预计费率）

### 6.2 下机流程
1. 员工选择进行中的机位，发起下机
2. 系统记录 `end_time = NOW()`
3. 计费引擎计算总时长（分钟）和费用
   - 查询 `rate_rules` 匹配该时间段所有规则
   - 按分钟切片计算每个时段费用
   - 如是会员，应用折扣等级
4. 生成账单摘要
5. 处理支付：
   - 会员余额支付 → 扣减余额并记录 `recharges` 流水（消费类型）
   - 现金/扫码 → 记录支付方式
6. 更新 `sessions` 状态为 `已结束`
7. 更新 `computers.status = 0`，清空 `current_session_id`
8. 更新 `members.total_spent` 累计消费

### 6.3 计费引擎算法（关键）
- 输入：开始时间、结束时间、机位单价、会员折扣
- 输出：总费用、费用明细（每个时段计费）
- 算法：
  1. 遍历开始到结束的每一分钟
  2. 每分钟查找匹配的 `rate_rule`（按 day_of_week + start_time/end_time）
  3. 如果有匹配规则，按规则单价计算；否则按机位默认 `hourly_rate`
  4. 汇总总金额
  5. 应用会员折扣（如有）
- 性能考虑：批量分钟循环可能较慢，可改为按时间段切片处理

### 6.4 并发冲突处理
- **同一机位双人上机**：在 Service 层加锁（GORM 的 `Clauses(clause.Locking{Strength: "UPDATE"})`）
- **会员余额不足上机**：上机时预检查 `balance >= 最低消费`（如 1 小时费用）
- **下机时余额不足**：允许欠费（记录负余额），下次充值补齐

## 附录：关键设计决策记录

1. **为什么会员用 card_number 而非自动递增 ID？** — 便于线下实体卡印刷和管理，可自定义编号规则。
2. **为什么 sessions 表冗余 current_session_id 到 computers？** — 减少联表查询，提升机位网格展示性能。
3. **计费规则为什么用优先级字段？** — 支持特殊时段覆盖（如节假日覆盖普通规则）。
4. **为什么下机时才计算费用？** — 避免频繁计算占用 CPU，且上机过程中可能修改规则，以下机时为准更符合业务。
5. **前端为什么不用 Websocket？** — 降低复杂度，轮询足以满足 50-100 台机位的实时性需求。
