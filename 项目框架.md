## 一、后端目录详解（backend/）

### 1.1 cmd/ - 应用程序入口

```
cmd/
└── server/
    └── main.go                       # 唯一的程序入口
```

#### `main.go` 职责
- **初始化顺序**：加载配置 → 连接数据库 → 依赖注入 → 注册路由 → 启动服务
- **优雅关闭**：监听系统信号（Ctrl+C），等待正在处理的请求完成后再退出
- **依赖组装**：这里会实例化所有的 Repository → Service → Handler，并传入路由

**开发注意**：
- main.go 应该保持简洁，只做"组装"工作，不要包含业务逻辑
- 错误处理：任何一个初始化步骤失败都应该 `log.Fatal` 立即退出

#### 1.2.2 domain/ - 领域模型

```
domain/
├── user.go
├── member.go
├── computer.go
├── session.go
├── rate_rule.go
├── recharge.go
├── audit_log.go
├── session_detail.go      # 用于上机记录详情的组合结构
└── statistics.go           # 统计报表的数据结构
```

**每个文件的职责**：
- 定义纯 Go 结构体，**只包含字段和 JSON 标签**
- 不包含 GORM 标签、不包含数据库操作、不包含业务方法
- 用于各层之间数据传输（DTO）

**示例（user.go）**：
```go
type User struct {
    ID          uint      `json:"id"`
    Username    string    `json:"username"`
    RealName    string    `json:"realName"`
    Role        int       `json:"role"`
    Status      int       `json:"status"`
    LastLoginAt time.Time `json:"lastLoginAt"`
    CreatedAt   time.Time `json:"createdAt"`
    UpdatedAt   time.Time `json:"updatedAt"`
}
```

**开发注意**：
- domain 与 model 的区别：domain 是"业务对象"，model 是"数据库映射对象"
- 当两者结构一致时，可以合并，但建议分开以便未来演进

**接口定义（interface/user_repo.go）**：

type UserRepository interface {
    Create(user *domain.User) error
    GetByID(id uint) (*domain.User, error)
    GetByUsername(username string) (*domain.User, error)

- Repository 只做 CRUD，不包含业务逻辑
- 返回 domain 对象而非 GORM 对象（解耦）
- 每个表对应一个 Repository 接口和实现

#### 1.2.4 service/ - 业务逻辑层

```
service/
├── auth_service.go                 # 认证服务（登录/登出/刷新Token）
├── computer_service.go             # 机位管理
├── session_service.go              # 上机/下机（核心业务）
├── member_service.go               # 会员管理
├── rate_service.go                 # 计费规则管理
├── recharge_service.go             # 充值业务
├── statistics_service.go           # 统计报表
├── audit_service.go                # 操作日志
├── billing_engine.go               # 计费引擎（核心算法）
└── service.go                      # 聚合所有 Service 的容器
```

**每个文件的职责**：

| 文件                    | 核心功能                            | 调用的 Repository                     |
| ----------------------- | ----------------------------------- | ------------------------------------- |
| `auth_service.go`       | 密码验证、JWT 生成/刷新、登出黑名单 | UserRepo                              |
| `computer_service.go`   | 机位 CRUD、状态切换、区域管理       | ComputerRepo                          |
| `session_service.go`    | 上机/下机事务编排、状态管理         | SessionRepo, ComputerRepo, MemberRepo |
| `member_service.go`     | 会员 CRUD、余额查询、折扣计算       | MemberRepo, RechargeRepo              |
| `rate_service.go`       | 计费规则 CRUD、时段冲突检测         | RateRuleRepo                          |
| `recharge_service.go`   | 充值操作、余额更新、流水记录        | RechargeRepo, MemberRepo              |
| `statistics_service.go` | 聚合查询、数据计算、报表生成        | SessionRepo, ComputerRepo             |
| `audit_service.go`      | 操作日志记录（异步或同步）          | AuditLogRepo                          |
| `billing_engine.go`     | 计费算法核心（时段匹配、费用计算）  | 直接使用 GORM 查询 RateRule           |

**Service 层核心要点**：
- **事务编排**：`session_service.go` 的上机方法需要同时操作多张表，必须用事务
- **计费引擎**：独立的 `billing_engine.go`，输入开始/结束时间和机位ID，输出费用明细
- **错误转换**：将 Repository 返回的数据库错误转换为业务错误码

**开发注意**：
- Service 之间可以相互调用（如 SessionService 调用 MemberService 的扣费方法）
- 所有业务错误应定义在 `service/errors.go` 中，用自定义错误类型

#### 1.2.5 handler/ - HTTP 控制器层

```
handler/
├── v1/                              # API v1 版本
│   ├── auth_handler.go
│   ├── computer_handler.go
│   ├── session_handler.go
│   ├── member_handler.go
│   ├── rate_handler.go
│   ├── recharge_handler.go
│   ├── statistics_handler.go
│   └── audit_handler.go
├── response.go                      # 统一响应封装
└── errors.go                        # HTTP 错误码定义
```

**每个 Handler 的职责**：
1. 从请求中提取参数（路径参数、查询参数、JSON 体）
2. 参数校验（使用 Gin 的 binding 或自定义校验器）
3. 调用对应的 Service 方法
4. 将 Service 返回的结果封装为统一响应格式

**示例（session_handler.go 的方法）**：
```go
// StartSession 上机
func (h *SessionHandler) StartSession(c *gin.Context) {
    var req StartSessionRequest
    if err := c.ShouldBindJSON(&req); err != nil {
        response.Error(c, ErrCodeInvalidParam, err.Error())
        return
    }
    
    session, err := h.sessionService.StartSession(req.ComputerID, req.MemberID)
    if err != nil {
        response.Error(c, ErrCodeBusinessError, err.Error())
        return
    }
    
    response.Success(c, session)
}
```

**开发注意**：
- Handler 应该非常薄，只做"参数-响应"的转换
- 不要在 Handler 里写业务逻辑或直接操作数据库
- 每个 Service 方法对应一个或多个 HTTP 接口

#### 1.2.6 middleware/ - 中间件

```
middleware/
├── auth.go               # JWT 验证中间件
├── cors.go               # 跨域处理
├── logger.go             # 请求日志记录
├── rate_limit.go         # 限流中间件
├── role_check.go         # RBAC 权限校验
├── recovery.go           # 错误恢复（panic 捕获）
└── trace.go              # 请求链路追踪（生成 TraceID）
```

**各中间件执行顺序**（按注册顺序）：
```
Recovery → Trace → Logger → CORS → RateLimit → Auth → RoleCheck → Handler
```

| 中间件          | 执行时机 | 核心功能                                  |
| --------------- | -------- | ----------------------------------------- |
| `recovery.go`   | 最外层   | 捕获 panic，返回 500 而非崩溃             |
| `trace.go`      | 第二层   | 生成 TraceID，存入 `c.Set()` 用于日志关联 |
| `logger.go`     | 第三层   | 记录请求开始/结束时间、状态码、耗时       |
| `cors.go`       | 第四层   | 处理 OPTIONS 预检，设置跨域头             |
| `rate_limit.go` | 第五层   | 基于 IP 或用户 ID 限制请求频率            |
| `auth.go`       | 第六层   | 验证 JWT，提取用户信息存入上下文          |
| `role_check.go` | 最内层   | 检查当前用户是否有权限访问该路由          |

**开发注意**：
- `auth.go` 中验证失败应返回 401，但不要 `Abort()` 后还执行后续逻辑
- `role_check.go` 需要从 `c.Get("user")` 获取用户信息
- 限流实现可基于 `golang.org/x/time/rate` 或 Redis 实现分布式限流

#### 1.2.7 router/ - 路由注册

```
router/
└── router.go            # 所有路由注册
```

**`router.go` 职责**：
1. 创建 Gin 引擎实例
2. 注册全局中间件
3. 注册健康检查接口 (`/health`, `/ping`)
4. 注册 API 路由组 (`/api/v1`)
5. 在路由组内注册各 Handler 的路由

**路由组织示例**：
```go
func SetupRouter(
    authHandler *handler.AuthHandler,
    computerHandler *handler.ComputerHandler,
    // ... 其他 handler
) *gin.Engine {
    r := gin.Default()
    
    // 全局中间件
    r.Use(middleware.CORS())
    r.Use(middleware.Trace())
    
    // 健康检查（无需认证）
    r.GET("/health", healthCheck)
    
    // API v1
    v1 := r.Group("/api/v1")
    {
        // 认证组（无需认证）
        auth := v1.Group("/auth")
        {
            auth.POST("/login", authHandler.Login)
            auth.POST("/refresh", authHandler.RefreshToken)
        }
        
        // 需要认证的接口
        authorized := v1.Group("/")
        authorized.Use(middleware.Auth())
        {
            authorized.GET("/profile", authHandler.Profile)
            
            // 机位管理
            computers := authorized.Group("/computers")
            {
                computers.GET("", computerHandler.List)
                computers.POST("", computerHandler.Create)
                computers.PUT("/:id", computerHandler.Update)
                computers.DELETE("/:id", computerHandler.Delete)
            }
            
            // 需要管理员权限
            admin := authorized.Group("/admin")
            admin.Use(middleware.RoleCheck(2))
            {
                admin.GET("/audit-logs", auditHandler.List)
            }
        }
    }
    
    return r
}
```

**开发注意**：
- 路由注册函数接收所有 Handler 的依赖
- 按模块分组，保持清晰
- 每个 HTTP 方法 + 路径唯一对应一个 Handler 方法

#### 1.2.8 model/ - ORM 模型

```
model/
├── gorm_models.go        # GORM 模型定义（所有表）
└── migrations.go         # 自动迁移（可选）
```

**`gorm_models.go` 内容**：
- 定义与数据库表结构完全对应的 GORM 结构体
- 包含 GORM 标签（`gorm:"..."`）
- 包含字段注释（用于生成表注释）

**示例**：
```go
type GormUser struct {
    ID          uint      `gorm:"primaryKey"`
    Username    string    `gorm:"uniqueIndex;size:50;not null;comment:用户名"`
    PasswordHash string   `gorm:"column:password_hash;size:255;not null;comment:加密密码"`
    RealName    string    `gorm:"size:50;comment:真实姓名"`
    Role        int       `gorm:"default:1;comment:1员工 2管理员 3超管"`
    Status      int       `gorm:"default:1;comment:1启用 0禁用"`
    LastLoginAt *time.Time `gorm:"comment:最后登录时间"`
    CreatedAt   time.Time `gorm:"autoCreateTime"`
    UpdatedAt   time.Time `gorm:"autoUpdateTime"`
}

func (GormUser) TableName() string {
    return "users"
}
```

**开发注意**：
- 如果 domain 和 model 完全一致，可以合并
- 迁移脚本建议用 SQL 文件手动管理，而非 GORM AutoMigrate（生产环境更可控）

#### 2.2.7 composables/ - 组合式函数

```
composables/
├── useAuth.js               # 认证逻辑（登录/登出/检查权限）
├── usePolling.js            # 轮询逻辑
├── useTimer.js              # 计时器（上机时长显示）
├── useTablePagination.js    # 表格分页逻辑

├── useConfirm.js            # 确认对话框
└── useNotification.js       # 通知提示
```

**每个 Composable 的职责**：

| 文件                    | 功能                                 | 返回值                                  |
| ----------------------- | ------------------------------------ | --------------------------------------- |
| `useAuth.js`            | 封装登录、登出、权限检查逻辑         | `{ login, logout, checkPermission }`    |
| `usePolling.js`         | 定时执行函数，自动清理               | `{ start, stop, isActive }`             |
| `useTimer.js`           | 倒计时/计时显示（格式化为 HH:mm:ss） | `{ formattedTime, start, stop, reset }` |
| `useTablePagination.js` | 表格分页、排序、筛选逻辑             | `{ page, pageSize, total, loadData }`   |
| `useDebounce.js`        | 防抖处理搜索输入                     | `{ debouncedValue, setValue }`          |
| `useConfirm.js`         | 封装确认对话框（返回 Promise）       | `{ confirm }`                           |
| `useNotification.js`    | 封装消息通知（成功/错误/警告）       | `{ success, error, warning }`           |

**示例（usePolling.js）**：
```javascript
export function usePolling(fn, interval = 5000) {
  const timer = ref(null)
  const isActive = ref(true)

  const start = () => {
    if (timer.value) return
    timer.value = setInterval(() => {
      if (isActive.value) fn()
    }, interval)
  }

  const stop = () => {
    clearInterval(timer.value)
    timer.value = null
  }

  onUnmounted(stop)
  return { start, stop, isActive }
}
```

**开发注意**：
- Composable 命名以 `use` 开头
- 内部使用 Vue 的响应式 API（ref/reactive/computed/watch/生命周期）
- 在组件中使用时，不需要手动清理（onUnmounted 已在内部处理）

## 六、常见问题 FAQ

### Q1: 为什么要有 domain 和 model 两个目录？
- **domain**：与数据库无关的业务对象，用于层间传输
- **model**：GORM 映射对象，包含数据库标签
- 当两者一致时可以合并，但分离更灵活（如未来切换 ORM）

### Q2: Repository 接口和实现为什么要分开？
- **接口**：定义契约，方便测试时 Mock
- **实现**：真正的数据库操作
- 依赖倒置原则：Service 依赖接口，不依赖具体实现

### Q3: pkg 和 internal 的区别？
- **internal**：只能被本项目内部导入，外部无法引用
- **pkg**：可以被外部项目导入（但本项目暂时不需要）
- 放在 internal 可以防止被其他项目误用

### Q4: 前端组件如何决定放在 views 还是 components？
- **views**：对应路由的页面级组件，包含页面布局
- **components**：可复用的组件，被 views 或其他 components 引用
- 如果某个组件只在一个页面使用，可以放在该页面的 `components/` 子目录

### Q5: Store 和 Composable 如何选择？
- **Store**：管理**全局共享**的状态（跨多个组件）
- **Composable**：封装**可复用逻辑**，不一定需要全局状态
- 例如：轮询逻辑用 Composable，用户信息用 Store

### Q6: 如何避免循环依赖？
- 遵循分层依赖方向：Handler → Service → Repository → Model
- 同层之间谨慎引用（如 Service 之间可以互相调用，但要避免循环）
- 使用依赖注入，不要在包内直接 `import` 具体实现
